# 分类诊断问题 (Triage Issue)

调查报告的问题，找出其根本原因，并创建一个包含 TDD 修复计划的 GitHub Issue。这主要是一个无需过多干预的工作流程——尽量减少对用户的提问。

## 流程

### 1. 捕获问题

从用户处获取关于问题的简要描述。如果他们没有提供，只问**一个**问题：“您遇到了什么问题？”

暂时**不要**问后续问题。立即开始调查。

### 2. 探索与诊断

使用带有 `subagent_type=Explore` 的 Agent 工具深入调查代码库。你的目标是找出：

- **在哪里**表现出该缺陷（入口点、UI、API 响应）
- 涉及了**什么**代码路径（追踪执行流）
- **为什么**会失败（根本原因，而不仅仅是症状）
- 存在**哪些**相关代码（相似模式、测试、相邻模块）

查看：
- 相关的源文件及其依赖项
- 现有的测试（测试了什么，遗漏了什么）
- 受影响文件的最近更改（在相关文件上运行 `git log`）
- 代码路径中的错误处理
- 代码库中其他能够正常工作的相似模式

### 3. 确定修复方案

基于你的调查，确定：

- 修复根本原因所需的最小更改
- 哪些模块/接口会受到影响
- 需要通过测试验证哪些行为
- 这是一个回归问题、功能缺失还是设计缺陷

### 4. 设计 TDD 修复计划

创建一个具体的、有序的 RED-GREEN（红-绿）循环列表。每个循环是一个垂直切片：

- **RED（红）**：描述一个能捕获损坏/缺失行为的具体测试
- **GREEN（绿）**：描述能使该测试通过的最小代码更改

规则：
- 测试应通过公共接口验证行为，而不是验证实现细节
- 每次只处理一个测试，采用垂直切片方式（**切勿**先写完所有测试，再写所有代码）
- 每个测试都应能在内部重构后继续生效
- 如果需要，包含最后一步重构（refactor）
- **持久性**：只提供能够经受住代码库剧烈变化的修复建议。描述行为和契约，而不是内部结构。测试断言应针对可观察的结果（API 响应、UI 状态、用户可见的副作用），而不是内部状态。一个好的建议读起来像是一份规格说明书；一个糟糕的建议读起来像是代码差异对比（diff）。

### 5. 创建 GitHub Issue

使用 `gh issue create` 和下方的模板创建一个 GitHub Issue。**不要**在创建前要求用户审查——直接创建并分享 URL。

<issue-template>

## 问题描述

对缺陷或问题的清晰描述，包括：
- 发生了什么（实际行为）
- 应该发生什么（预期行为）
- 如何重现（如果适用）

## 根本原因分析

描述你在调查中发现的内容：
- 涉及的代码路径
- 当前代码失败的原因
- 任何促发因素

**不要**包含特定的文件路径、行号或与当前代码布局耦合的实现细节。相反，应该描述模块、行为和契约。即使在发生重大重构之后，该 Issue 也应保持其参考价值。

## TDD 修复计划

一个包含 RED-GREEN 循环的编号列表：

1. **RED**：编写一个测试，[描述预期行为]
   **GREEN**：[使其通过的最小更改]

2. **RED**：编写一个测试，[描述下一个行为]
   **GREEN**：[使其通过的最小更改]

...

**REFACTOR（重构）**：[所有测试通过后需要的任何清理工作]

## 验收标准

- [ ] 标准 1
- [ ] 标准 2
- [ ] 所有新测试均通过
- [ ] 现有测试仍然通过

</issue-template>

创建 Issue 后，输出 Issue URL 和一行根本原因的摘要。
