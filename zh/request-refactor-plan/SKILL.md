---

当用户想要创建重构请求时，将调用此技能。你应该执行以下步骤。如果你认为某些步骤没有必要，可以跳过它们。

1. 请用户详细、全面地描述他们想要解决的问题，以及任何潜在的解决方案想法。

2. 探索代码仓库以验证他们的断言，并了解代码库的当前状态。

3. 询问他们是否考虑过其他方案，并向他们提供其他方案。

4. 就实现细节采访用户。务必极其详尽和彻底。

5. 敲定实现的具体范围。明确计划更改的内容和不计划更改的内容。

6. 查看代码库，检查该区域代码的测试覆盖率。如果测试覆盖率不足，询问用户他们的测试计划是什么。

7. 将实现拆分为一个微小提交的计划。请记住马丁·福勒（Martin Fowler）的建议：“让每个重构步骤尽可能小，这样你才能始终看到程序在正常运行。”

8. 使用重构计划创建一个 GitHub Issue。Issue 描述请使用以下模板：

<refactor-plan-template>

## 问题陈述

从开发者的角度描述其面临的问题。

## 解决方案

从开发者的角度描述问题的解决方案。

## 提交计划

一份详尽、深入的实现计划。用通俗易懂的语言编写该计划，将实现拆分为尽可能微小的提交。每次提交都应使代码库保持在可运行的状态。

## 决策文档

已做出的实现决策列表。这可以包括：

- 将要构建/修改的模块
- 将要修改的模块接口
- 来自开发者的技术澄清
- 架构决策
- 模式更改
- API 契约
- 具体交互

不要包含具体的文件路径或代码片段。它们可能很快就会过时。

## 测试决策

已做出的测试决策列表。包括：

- 关于什么是优秀测试的描述（仅测试外部行为，而非实现细节）
- 将要测试哪些模块
- 测试的先前经验（即代码库中类似类型的测试）

## 超出范围

描述本次重构范围之外的事项。

## 补充说明（可选）

关于重构的任何补充说明。

</refactor-plan-template>
