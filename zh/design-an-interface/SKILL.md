# 设计接口 (Design an Interface)

基于《软件设计哲学》(A Philosophy of Software Design) 中的“设计两次 (Design It Twice)”原则：你的第一个想法往往不是最好的。请生成多个截然不同的设计，然后进行比较。

## 工作流程

### 1. 收集需求

在设计之前，需要明确：

- [ ] 这个模块解决什么问题？
- [ ] 调用者是谁？（其他模块、外部用户、测试）
- [ ] 核心操作有哪些？
- [ ] 有任何约束条件吗？（性能、兼容性、现有模式）
- [ ] 哪些细节应该隐藏在内部，哪些应该暴露？

提问：“这个模块需要做什么？谁会使用它？”

### 2. 生成设计（并行子智能体）

使用 Task 工具同时生成 3 个以上的子智能体。每个智能体必须提供一种**截然不同**的方案。

```text
每个子智能体的提示词模板：

为以下模块设计一个接口：[模块描述]

需求：[收集到的需求]

本次设计的约束条件：[为每个智能体分配一个不同的约束]
- 智能体 1：“最小化方法数量——目标是最多 1-3 个方法”
- 智能体 2：“最大化灵活性——支持多种用例”
- 智能体 3：“针对最常见的情况进行优化”
- 智能体 4：“从 [特定范式/库] 中获取灵感”

输出格式：
1. 接口签名（类型/方法）
2. 使用示例（调用者如何使用）
3. 该设计在内部隐藏了什么
4. 这种方案的权衡（优缺点）
```

### 3. 展示设计

展示每个设计，包含以下内容：

1. **接口签名** - 类型、方法、参数
2. **使用示例** - 调用者在实践中如何实际使用
3. **它隐藏了什么** - 保持在内部的复杂性

按顺序呈现各个设计，以便用户在比较之前能够充分理解每种方案。

### 4. 比较设计

在展示所有设计后，从以下几个方面进行比较：

- **接口简洁性**：方法更少、参数更简单
- **通用与专用**：灵活性与专注度
- **实现效率**：接口形态是否允许高效的内部实现？
- **深度**：隐藏了显著复杂性的小接口（好） vs 只有单薄实现的大接口（坏）
- **正确使用的容易程度** vs **误用的容易程度**

用文字段落（而非表格）来讨论这些权衡。突出设计之间分歧最大的地方。

### 5. 综合

通常，最好的设计会结合多种方案的见解。提问：

- “哪种设计最适合你的主要用例？”
- “其他设计中有哪些元素值得借鉴和整合？”

## 评估标准

摘自《软件设计哲学》：

**接口简洁性**：方法更少、参数更简单 = 更容易学习和正确使用。

**通用性**：无需更改即可处理未来的用例。但要警惕过度通用化。

**实现效率**：接口形态是否允许高效的实现？还是会导致别扭的内部结构？

**深度**：隐藏显著复杂性的小接口 = 深模块（好）。具有单薄实现的大接口 = 浅模块（应避免）。

## 反模式

- 不要让子智能体产生相似的设计——必须强制要求截然不同
- 不要跳过比较环节——设计的价值就在于对比
- 不要进行具体实现——这纯粹是关于接口形态的讨论
- 不要基于实现的工作量来进行评估
