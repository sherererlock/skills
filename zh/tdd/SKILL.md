# 测试驱动开发 (Test-Driven Development)

## 理念

**核心原则**：测试应该通过公共接口验证行为，而不是验证实现细节。代码可以完全改变；而测试不应该改变。

**优秀的测试**是集成风格的：它们通过公共 API 运行真实的代码路径。它们描述系统_做_什么，而不是_怎么_做。一个好的测试读起来就像一份规格说明书——“用户可以使用有效的购物车结账”确切地告诉你存在什么功能。这些测试能够在重构中存活下来，因为它们不关心内部结构。

**糟糕的测试**与实现相耦合。它们模拟（mock）内部协作者，测试私有方法，或通过外部手段进行验证（比如直接查询数据库而不是使用接口）。警告信号：当你进行重构时测试失败了，但行为并没有改变。如果你重命名了一个内部函数导致测试失败，那么这些测试就是在测试实现，而不是行为。

有关示例，请参见 [tests.md](tests.md)；有关模拟（mocking）指南，请参见 [mocking.md](mocking.md)。

## 反模式：水平切片 (Horizontal Slices)

**不要先写完所有测试，然后再写所有实现。** 这被称为“水平切片”——将 RED 视为“编写所有测试”，将 GREEN 视为“编写所有代码”。

这会产生**垃圾测试**：

- 批量编写的测试验证的是_想象中的_行为，而不是_实际的_行为
- 你最终测试的是事物的_形状_（数据结构、函数签名），而不是面向用户的行为
- 测试对实际的变化变得不敏感——当行为遭到破坏时它们通过，当行为正常时它们反而失败
- 你的步伐超出了认知范围，在理解实现之前就过早地确定了测试结构

**正确的方法**：通过曳光弹（tracer bullets）进行垂直切片。一个测试 → 一个实现 → 重复。每个测试都在响应你从上一个循环中学到的东西。因为你刚刚编写了代码，你确切地知道哪些行为很重要以及如何验证它。

```text
错误 (水平切片):
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

正确 (垂直切片):
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  RED→GREEN: test3→impl3
  ...
```

## 工作流程

### 1. 计划

在编写任何代码之前：

- [ ] 与用户确认需要哪些接口变更
- [ ] 与用户确认要测试哪些行为（排定优先级）
- [ ] 寻找应用[深模块](deep-modules.md)（接口简单，实现深入）的机会
- [ ] 为[可测试性](interface-design.md)设计接口
- [ ] 列出要测试的行为（而不是实现步骤）
- [ ] 获得用户对计划的批准

询问：“公共接口应该是什么样的？哪些行为是最需要测试的？”

**你不可能测试所有东西。** 与用户确切确认哪些行为最重要。将测试精力集中在关键路径和复杂逻辑上，而不是每一个可能的边缘情况。

### 2. 曳光弹 (Tracer Bullet)

编写**一个**测试，确认关于系统的**一件事**：

```text
RED:   为第一个行为编写测试 → 测试失败
GREEN: 编写最少量的代码使其通过 → 测试通过
```

这就是你的曳光弹——证明这条路径端到端是通的。

### 3. 增量循环

对于每一个剩余的行为：

```text
RED:   编写下一个测试 → 失败
GREEN: 编写最少量的代码使其通过 → 通过
```

规则：

- 每次只写一个测试
- 只写刚好能让当前测试通过的代码
- 不要预判未来的测试
- 保持测试专注于可观察的行为

### 4. 重构

在所有测试通过后，寻找[重构候选项](refactoring.md)：

- [ ] 消除重复代码
- [ ] 加深模块（将复杂性隐藏在简单的接口背后）
- [ ] 在自然适用的地方应用 SOLID 原则
- [ ] 思考新代码对现有代码的启示
- [ ] 在每个重构步骤后运行测试

**绝不要在 RED 状态下进行重构。** 必须先回到 GREEN 状态。

## 每次循环的检查清单

```text
[ ] 测试描述的是行为，而不是实现
[ ] 测试仅使用公共接口
[ ] 测试能够在内部重构中存活下来
[ ] 针对此测试的代码是最少量的
[ ] 没有添加推测性的功能
```
