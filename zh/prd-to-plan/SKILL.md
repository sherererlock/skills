# PRD 转计划

使用垂直切片（曳光弹）将 PRD 拆解为分阶段的实施计划。输出结果为 `./plans/` 目录下的 Markdown 文件。

## 流程

### 1. 确认 PRD 已在上下文中

PRD 应该已经存在于对话上下文中。如果不存在，请要求用户粘贴它或向你指出该文件的位置。

### 2. 探索代码库

如果你还没有探索过代码库，请先进行探索，以了解当前的架构、现有的模式和集成层。

### 3. 确定持久的架构决策

在进行切片之前，请确定在整个实施过程中不太可能改变的高层决策：

- 路由结构 / URL 模式
- 数据库结构形态
- 关键数据模型
- 身份验证 / 授权方案
- 第三方服务边界

这些内容将放在计划的头部，以便每个阶段都可以参考它们。

### 4. 草拟垂直切片

将 PRD 拆解为多个**曳光弹**阶段。每个阶段都是一个穿透所有集成层的端到端薄垂直切片，而不是某个单层的水平切片。

<vertical-slice-rules>
- 每个切片都提供一条狭窄但完整的路径，贯穿所有层（数据库结构、API、UI、测试）
- 已完成的切片可以独立演示或验证
- 宁可采用许多薄切片，也不要使用少数厚切片
- 不要包含随着后续阶段构建可能发生变化的具体文件名、函数名或实施细节
- 必须包含持久的决策：路由路径、数据库结构形态、数据模型名称
</vertical-slice-rules>

### 5. 向用户提问

将建议的拆解方案以编号列表的形式呈现。对于每个阶段，请展示：

- **标题**：简短的描述性名称
- **涵盖的用户故事**：该阶段解决了 PRD 中的哪些用户故事

询问用户：

- 粒度感觉合适吗？（太粗 / 太细）
- 是否有任何阶段需要合并或进一步拆分？

迭代这一过程，直到用户批准拆解方案。

### 6. 编写计划文件

如果 `./plans/` 目录不存在，请创建它。将计划编写为一个 Markdown 文件，以功能命名（例如 `./plans/user-onboarding.md`）。使用以下模板。

<plan-template>
# 计划：<功能名称>

> 来源 PRD：<简短标识符或链接>

## 架构决策

适用于所有阶段的持久决策：

- **路由**：...
- **数据库结构**：...
- **关键模型**：...
- (根据需要添加/删除小节)

---

## 阶段 1：<标题>

**用户故事**：<来自 PRD 的列表>

### 要构建的内容

对该垂直切片的简明描述。描述端到端的行为，而不是逐层的实施细节。

### 验收标准

- [ ] 标准 1
- [ ] 标准 2
- [ ] 标准 3

---

## 阶段 2：<标题>

**用户故事**：<来自 PRD 的列表>

### 要构建的内容

...

### 验收标准

- [ ] ...

<!-- 对每个阶段重复上述结构 -->
</plan-template>
