name: write-a-prd
description: 通过用户访谈、代码库探索和模块设计来创建 PRD，然后作为 GitHub Issue 提交。当用户想要编写 PRD、创建产品需求文档或规划新功能时使用。
---

当用户想要创建 PRD 时，将调用此技能。如果你认为某些步骤没有必要，可以跳过。

1. 向用户询问他们想要解决的问题的详细长篇描述，以及任何潜在的解决方案想法。

2. 探索代码仓库以验证他们的想法，并了解代码库的当前状态。

3. 就该计划的各个方面与用户进行深入细致的沟通，直到你们达成共识。沿着设计树的每个分支深入探讨，逐一解决决策之间的依赖关系。

4. 勾勒出为了完成功能实现而需要构建或修改的主要模块。积极寻找提取“深层模块（deep modules）”的机会，使其可以在隔离状态下进行测试。

深层模块（与浅层模块相对）是指将大量功能封装在一个简单、可测试且极少更改的接口中的模块。

与用户确认这些模块的设计是否符合他们的期望。与用户确认他们希望为哪些模块编写测试。

5. 一旦你对问题和解决方案有了完整的理解，请使用下面的模板编写 PRD。该 PRD 应该作为一个 GitHub Issue 提交。

<prd-template>

## 问题陈述

从用户视角出发，用户所面临的问题。

## 解决方案

从用户视角出发，针对该问题的解决方案。

## 用户故事

一份极其详尽的、带编号的用户故事列表。每个用户故事应采用以下格式：

1. 作为一个 <actor/角色>，我想要 <feature/功能>，以便 <benefit/收益>

<user-story-example>
1. 作为一个手机银行客户，我希望看到我的账户余额，以便我能对我的消费做出更明智的决定
</user-story-example>

这份用户故事列表应该极其详尽，并涵盖该功能的所有方面。

## 实现决策

所做出的实现决策列表。这可以包括：

- 将要构建/修改的模块
- 这些模块中将要修改的接口
- 来自开发者的技术澄清
- 架构决策
- 数据模式 (Schema) 更改
- API 契约
- 具体交互

不要包含特定的文件路径或代码片段。它们可能会很快过时。

## 测试决策

所做出的测试决策列表。包括：

- 描述什么是好的测试（只测试外部行为，而不是实现细节）
- 哪些模块将被测试
- 测试的现有参考（即代码库中类似类型的测试）

## 超出范围

描述不在此 PRD 范围之内的事项。

## 补充说明

关于该功能的任何补充说明。

</prd-template>
