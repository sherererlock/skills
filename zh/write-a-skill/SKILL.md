# 编写技能

## 流程

1. **收集需求** - 询问用户：
   - 该技能涵盖什么任务/领域？
   - 它应该处理哪些具体用例？
   - 它需要可执行脚本还是仅仅是指令？
   - 有任何需要包含的参考资料吗？

2. **起草技能** - 创建：
   - 带有简明指令的 SKILL.md
   - 额外的参考文件（如果内容超过 500 行）
   - 实用脚本（如果需要确定性操作）

3. **与用户一起审查** - 展示草案并询问：
   - 这是否涵盖了您的用例？
   - 有什么遗漏或不清楚的地方吗？
   - 是否有任何部分需要更详细/更简略？

## 技能结构

```
skill-name/
├── SKILL.md           # 主要指令（必需）
├── REFERENCE.md       # 详细文档（视需要而定）
├── EXAMPLES.md        # 用法示例（视需要而定）
└── scripts/           # 实用脚本（视需要而定）
    └── helper.js
```

## SKILL.md 模板

```md
---
name: skill-name
description: 功能的简要描述。当[特定触发条件]时使用。
---

# 技能名称

## 快速开始

[最小可行示例]

## 工作流

[针对复杂任务的包含检查表的逐步流程]

## 高级功能

[链接到独立文件：参见 [REFERENCE.md](REFERENCE.md)]
```

## 描述要求

描述是**您的智能体在决定加载哪个技能时唯一会看到的内容**。它会与所有其他已安装的技能一起显示在系统提示词中。您的智能体会阅读这些描述，并根据用户的请求选择相关的技能。

**目标**：为您的智能体提供刚好足够的信息以了解：

1. 该技能提供什么功能
2. 何时/为何触发它（特定关键字、上下文、文件类型）

**格式**：

- 最多 1024 个字符
- 以第三人称编写
- 第一句：它的作用是什么
- 第二句：“当[特定触发条件]时使用”

**好的示例**：

```
从 PDF 文件中提取文本和表格、填写表单、合并文档。当处理 PDF 文件或当用户提及 PDF、表单或文档提取时使用。
```

**坏的示例**：

```
帮助处理文档。
```

坏的示例让您的智能体无法将其与其他文档技能区分开来。

## 何时添加脚本

在以下情况下添加实用脚本：

- 操作是确定性的（如验证、格式化）
- 会重复生成相同的代码
- 错误需要显式处理

与生成的代码相比，脚本可以节省 Token 并提高可靠性。

## 何时拆分文件

在以下情况下拆分为独立文件：

- SKILL.md 超过 100 行
- 内容具有不同的领域（如财务模式与销售模式）
- 很少需要高级功能

## 审查检查表

起草后，请验证：

- [ ] 描述包含触发条件（“当...时使用”）
- [ ] SKILL.md 不超过 100 行
- [ ] 没有时效性信息
- [ ] 术语一致
- [ ] 包含具体的示例
- [ ] 引用深度为一层
