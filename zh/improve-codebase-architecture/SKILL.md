# 改进代码库架构 (Improve Codebase Architecture)

像 AI 一样探索代码库，揭示架构上的摩擦，发现提高可测试性的机会，并将深化模块的重构提议作为 GitHub issue RFC 提出。

**深模块 (deep module)**（John Ousterhout，《软件设计的哲学》）拥有一个小的接口，却隐藏着庞大的实现。深模块更易于测试，更易于 AI 导航，并且允许你在边界进行测试，而不是在内部测试。

## 流程

### 1. 探索代码库

使用带有 `subagent_type=Explore` 参数的 Agent 工具自然地导航代码库。不要遵循死板的启发式规则——而是有机地探索并注意你在哪里遇到了阻力（friction）：

- 哪里需要你在许多小文件之间来回跳转才能理解一个概念？
- 哪里的模块太浅，以至于接口几乎和实现一样复杂？
- 哪里仅仅为了可测试性而提取了纯函数，但真正的 bug 却隐藏在它们被调用的方式中？
- 哪里紧密耦合的模块在它们之间的接缝处产生了集成风险？
- 代码库的哪些部分未经过测试，或者难以测试？

你遇到的阻力**就是**信号。

### 2. 提出候选方案

提供一个编号的深化机会列表。对于每个候选方案，展示：

- **集群 (Cluster)**：涉及哪些模块/概念
- **它们为何耦合 (Why they're coupled)**：共享类型、调用模式、共同拥有某个概念
- **依赖类别 (Dependency category)**：查看 [REFERENCE.md](REFERENCE.md) 了解四种类别
- **测试影响 (Test impact)**：哪些现有的测试将被边界测试所取代

**不要**急于提出接口设计。询问用户：“您想探索其中的哪一个？”

### 3. 用户选择候选方案

### 4. 框定问题空间

在生成子代理 (sub-agents) 之前，为所选候选方案写一段面向用户的关于问题空间的解释：

- 任何新接口都需要满足的约束条件
- 它需要依赖的项
- 一个粗略的说明性代码草图，以使约束条件具体化——这不是一个提案，只是为了让约束条件落地

向用户展示这些内容，然后立即进入第 5 步。在子代理并行工作的同时，用户阅读并思考这个问题。

### 5. 设计多个接口

使用 Agent 工具并行生成 3 个以上的子代理。每个子代理必须为深化后的模块生成一个**截然不同**的接口。

为每个子代理提供一份单独的技术简报（文件路径、耦合细节、依赖类别、隐藏的内容）。这份简报独立于第 4 步中面向用户的解释。给每个代理一个不同的设计约束：

- 代理 1：“最小化接口——目标是最多 1-3 个入口点”
- 代理 2：“最大化灵活性——支持许多用例和扩展”
- 代理 3：“针对最常见的调用者进行优化——使默认情况变得简单”
- 代理 4（如适用）：“围绕跨边界依赖的端口和适配器 (ports & adapters) 模式进行设计”

每个子代理输出：

1. 接口签名（类型、方法、参数）
2. 用法示例，展示调用者如何使用它
3. 它在内部隐藏了什么复杂性
4. 依赖策略（如何处理依赖项——参见 [REFERENCE.md](REFERENCE.md)）
5. 权衡取舍 (Trade-offs)

按顺序展示设计方案，然后用文字进行比较。

比较之后，给出你自己的建议：你认为哪个设计最强，为什么。如果不同设计中的元素结合起来会很好，就提出一个混合方案。要有主见——用户希望看到有力的见解，而不仅仅是一份菜单。

### 6. 用户选择一个接口（或接受建议）

### 7. 创建 GitHub issue

使用 `gh issue create` 创建一个重构 RFC 作为 GitHub issue。使用 [REFERENCE.md](REFERENCE.md) 中的模板。在创建之前**不要**要求用户进行审查——只需创建它并分享 URL。
