# 参考指南

## 依赖分类

在评估深化（deepening）候选模块时，请对其依赖进行分类：

### 1. 进程内依赖 (In-process)

纯计算、内存状态、无 I/O。总是可以被深化——只需合并这些模块并直接进行测试。

### 2. 本地可替代依赖 (Local-substitutable)

具有本地测试替代品的依赖（例如：用 PGLite 替代 Postgres，或内存文件系统）。如果存在测试替代品，则可以进行深化。深化后的模块通过在测试套件中运行本地替代品来进行测试。

### 3. 远程但受控依赖 (Ports & Adapters)

跨越网络边界的自有服务（微服务、内部 API）。在模块边界处定义一个端口（接口）。深模块拥有核心逻辑；传输层则是被注入的。测试环境使用内存适配器，生产环境使用真实的 HTTP/gRPC/队列适配器。

建议形式：“定义一个共享接口（端口），为生产环境实现 HTTP 适配器，为测试环境实现内存适配器，这样即使逻辑部署跨越了网络边界，它也可以作为一个深模块进行整体测试。”

### 4. 真正的外部依赖 (Mock)

你不受控的第三方服务（如 Stripe、Twilio 等）。在边界处进行 Mock（模拟）。深化后的模块将外部依赖作为注入的端口，测试环境提供 Mock 实现。

## 测试策略

核心原则：**替换，而不是叠加。**

- 一旦边界测试建立，浅模块的旧单元测试就成了浪费——删除它们
- 在深化后的模块接口边界处编写新的测试
- 测试应通过公共接口断言可观察的结果，而不是断言内部状态
- 测试应该能在内部重构后继续有效——它们描述的是行为，而不是实现

## Issue 模板

<issue-template>

## 问题描述

描述架构上的摩擦点：

- 哪些模块过于浅显且紧密耦合
- 它们之间的接缝处存在什么集成风险
- 为什么这使得代码库更难导航和维护

## 提议的接口

所选的接口设计：

- 接口签名（类型、方法、参数）
- 展示调用方如何使用的示例
- 它在内部隐藏了哪些复杂性

## 依赖策略

适用哪个分类以及如何处理依赖：

- **进程内依赖**：直接合并
- **本地可替代依赖**：使用 [特定的替代品] 进行测试
- **端口和适配器**：端口定义、生产适配器、测试适配器
- **Mock**：针对外部服务的 Mock 边界

## 测试策略

- **需要编写的新边界测试**：描述在接口处需要验证的行为
- **需要删除的旧测试**：列出变得多余的浅模块测试
- **测试环境需求**：任何需要的本地替代品或适配器

## 实现建议

不与当前文件路径耦合的持久架构指导原则：

- 模块应该拥有什么（职责）
- 它应该隐藏什么（实现细节）
- 它应该暴露什么（接口契约）
- 调用方应该如何迁移到新接口
