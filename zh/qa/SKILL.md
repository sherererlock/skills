# QA Session (QA 会话)

运行一个交互式 QA 会话。由用户描述他们遇到的问题。你需要澄清问题，在后台探索代码库以获取上下文，并提交持久的、以用户为中心的、使用项目领域语言的 GitHub issue。

## 针对用户提出的每个问题

### 1. 倾听并适度澄清

让用户用自己的话来描述问题。最多问 **2-3 个简短的澄清问题**，重点关注：

- 他们的期望是什么，而实际发生了什么
- 复现步骤（如果不够明显）
- 问题是持续存在的还是偶发的

**不要**过度盘问。如果描述已经足够清晰可以提交 issue，就继续下一步。

### 2. 在后台探索代码库

在与用户交谈时，在后台启动一个智能体（subagent_type=Explore）来了解相关领域。目标**不是**找到修复方案，而是：

- 学习该领域使用的领域语言（查看 UBIQUITOUS_LANGUAGE.md）
- 了解该功能应该做什么
- 识别面向用户的行为边界

这些上下文有助于你编写更好的 issue——但 issue 本身**不应**引用特定文件、行号或内部实现细节。

### 3. 评估范围：单个 issue 还是拆分？

在提交之前，决定这是一个**单个 issue** 还是需要**拆分**成多个 issue。

在以下情况下进行拆分：

- 修复涉及多个独立的区域（例如：“表单验证错误，且缺少成功提示，同时重定向也坏了”）
- 存在明显可分离的关注点，不同的人可以并行处理
- 用户描述的情况有多种截然不同的故障模式或症状

在以下情况下保持为单个 issue：

- 这是一个在同一位置发生的错误行为
- 所有的症状都是由同一个根本行为引起的

### 4. 提交 GitHub issue

使用 `gh issue create` 创建 issue。**不要**先让用户审核——直接提交并分享 URL 即可。

Issue 必须是**持久的**——它们在重大重构之后应该仍然有意义。请从用户的角度来编写。

#### 对于单个 issue

使用此模板：

```
## What happened (发生了什么)

[用通俗的语言描述用户经历的实际行为]

## What I expected (我的期望)

[描述期望的行为]

## Steps to reproduce (复现步骤)

1. [开发者可以遵循的具体的、带编号的步骤]
2. [使用代码库中的领域术语，而不是内部模块名称]
3. [包括相关的输入、标志或配置]

## Additional context (补充上下文)

[任何来自用户或代码库探索的额外观察结果，有助于界定问题——例如“这只在使用 Docker 层时发生，在文件系统层不发生”——使用领域语言，但不要引用文件]
```

#### 对于拆分的情况（多个 issue）

按依赖顺序（阻塞项优先）创建 issue，以便你可以引用真实的 issue 编号。

为每个子 issue 使用此模板：

```
## Parent issue (父级 issue)

#<父级-issue-编号>（如果你创建了跟踪 issue）或“在 QA 会话期间报告 (Reported during QA session)”

## What's wrong (出了什么问题)

[描述这个特定的行为问题——只描述这部分，而不是整个报告]

## What I expected (我的期望)

[针对这个特定部分的期望行为]

## Steps to reproduce (复现步骤)

1. [特定于此 issue 的步骤]

## Blocked by (前置依赖 / 被什么阻塞)

- #<issue-编号>（如果必须在另一个 issue 解决后才能修复此 issue）

如果没有阻塞项，则填写“None — can start immediately (无——可以立即开始)”。

## Additional context (补充上下文)

[任何与此部分相关的额外观察结果]
```

创建拆分时：

- **宁可多建小而精的 issue，也不要少建大而全的 issue**——每一个都应该是可独立修复和验证的
- **如实标记阻塞关系**——如果 issue B 确实在 issue A 修复之前无法测试，请如实说明。如果它们是独立的，请将两者都标记为“None — can start immediately (无——可以立即开始)”
- **按依赖顺序创建 issue**，这样你就可以在“Blocked by”中引用真实的 issue 编号
- **最大化并行性**——目标是多个人（或智能体）可以同时处理不同的 issue

#### 所有 issue 正文的规则

- **不要写文件路径或行号**——这些会过时
- **使用项目的领域语言**（如果存在，请查看 UBIQUITOUS_LANGUAGE.md）
- **描述行为，而不是代码**——例如写“同步服务无法应用补丁 (the sync service fails to apply the patch)”，而不是“applyPatch() 在第 42 行抛出异常 (applyPatch() throws on line 42)”
- **复现步骤是强制性的**——如果你无法确定它们，请询问用户
- **保持简洁**——开发者应该能够在 30 秒内读完这个 issue

提交后，打印所有 issue 的 URL（总结阻塞关系）并询问：“下一个 issue，还是我们已经完成了？”

### 5. 继续会话

继续进行，直到用户说他们完成了。每个 issue 都是独立的——不要将它们批量处理。
