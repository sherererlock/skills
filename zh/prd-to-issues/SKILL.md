# 将 PRD 拆分为 Issue

使用垂直切片（曳光弹）将 PRD 拆分为可独立领取的 GitHub issue。

## 流程

### 1. 定位 PRD

向用户询问 PRD 的 GitHub issue 编号（或 URL）。

如果 PRD 尚未在您的上下文窗口中，请使用 `gh issue view <number>`（包含评论）获取它。

### 2. 探索代码库（可选）

如果您尚未探索过代码库，请进行探索以了解代码的当前状态。

### 3. 起草垂直切片

将 PRD 拆分为**曳光弹** issue。每个 issue 都是一个薄薄的垂直切片，端到端地贯穿所有集成层，而不是单层的水平切片。

切片可以分为“HITL”（Human-in-the-loop，需要人工参与）或“AFK”（Away from keyboard，无需人工参与）。HITL 切片需要人工交互，例如架构决策或设计审查。AFK 切片可以在没有人工交互的情况下实现并合并。尽可能优先选择 AFK 而不是 HITL。

<vertical-slice-rules>
- 每个切片都在每一层（模式、API、UI、测试）中提供一条狭窄但完整的路径
- 完成的切片可以独立演示或验证
- 宁可切片多而薄，不要切片少而厚
</vertical-slice-rules>

### 4. 询问用户

将提议的拆分方案以编号列表的形式呈现。对于每个切片，请显示：

- **标题**：简短的描述性名称
- **类型**：HITL / AFK
- **前置依赖**：必须先完成哪些其他切片（如果有）
- **涵盖的用户故事**：这解决了 PRD 中的哪些用户故事

询问用户：

- 粒度感觉合适吗？（太粗 / 太细）
- 依赖关系正确吗？
- 是否应该合并或进一步拆分任何切片？
- 是否正确地将切片标记为了 HITL 和 AFK？

不断迭代，直到用户批准该拆分方案。

### 5. 创建 GitHub issue

对于每个获得批准的切片，使用 `gh issue create` 创建一个 GitHub issue。请使用下方的 issue 正文模板。

按照依赖顺序（阻塞项优先）创建 issue，以便您可以在“前置依赖”字段中引用真实的 issue 编号。

<issue-template>
## 父级 PRD

#<prd-issue-number>

## 构建内容

对此垂直切片的简明描述。描述端到端行为，而不是逐层实现。引用父级 PRD 的特定部分，而不是复制内容。

## 验收标准

- [ ] 标准 1
- [ ] 标准 2
- [ ] 标准 3

## 前置依赖

- 被 #<issue-number> 阻塞（如果有）

如果没有阻塞项，则填写“无 - 可以立即开始”。

## 解决的用户故事

通过编号引用父级 PRD：

- 用户故事 3
- 用户故事 7

</issue-template>

请勿关闭或修改父级 PRD issue。
