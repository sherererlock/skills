## improve-codebase-architecture：让代码库从"浅"变"深"的架构重构指南

这个技能的核心使命只有一句话：**把浅模块变成深模块**。它借鉴 Ousterhout《A Philosophy of Software Design》的"深度"理论，为 AI 辅助的代码库架构改进提供了一套完整的操作框架。

### 技能定位

它不是通用的重构工具，而是专门针对"架构摩擦"的诊断与治疗。触发场景是：代码库难以理解、难以测试、模块之间耦合混乱、AI 在导航时频繁迷路。它的目标是让代码库对人和 AI 都更友好——**可测试性**和 **AI 可导航性**是并列的一等公民。

### 核心设计哲学：深度即杠杆

技能引入了一套严格的词汇体系（LANGUAGE.md），要求在所有建议中精确使用：**模块、接口、实现、深度、接缝、适配器、杠杆、局部性**。这不是形式主义，而是刻意设计——统一语言能让 AI 和人类在同一概念框架下对话，避免"组件""服务""边界"等模糊词汇带来的歧义。

**深度**的定义是：接口小，行为多。一个深模块用极小的接口隐藏大量实现复杂度，调用者获得高杠杆；一个浅模块的接口复杂度几乎等于实现复杂度，调用者得不到任何简化。

### 关键机制细节

**删除测试（Deletion Test）** 是识别浅模块的核心工具：想象删掉这个模块，复杂度是消失了还是分散到 N 个调用者？如果是后者，说明它在"赚取"自己的存在；如果是前者，它只是一个透传层。

**依赖分类**（DEEPENING.md）决定了如何跨接缝测试深化后的模块：
- 纯内存计算 → 直接合并，无需适配器
- 有本地替代品（如 PGLite）→ 用替代品在测试套件中运行
- 自有远程服务 → 定义端口接口，生产用 HTTP 适配器，测试用内存适配器
- 第三方服务 → 注入端口，测试用 Mock 适配器

**"一个适配器 = 假接缝，两个适配器 = 真接缝"** 是防止过度设计的护栏——不要为了"灵活性"引入只有一个实现的接口，那只是无意义的间接层。

**接口设计**（INTERFACE-DESIGN.md）采用"设计两次"原则：并行启动 3+ 个子 Agent，分别从"最小接口""最大灵活性""最优化常见调用者"等不同约束角度各自设计接口，再汇总比较。这是对抗"第一个想法就是最好想法"偏见的系统性对策。

**测试策略**强调"替换而非叠加"：旧的浅模块单元测试在深化后应当删除，新测试写在深模块的接口上，断言可观测行为而非内部状态，且能在实现重构后存活。

### 亮点总结

这个技能最精妙的地方在于它的**副作用设计**：在与用户的讨论过程中，如果命名了一个新概念，立即写入 CONTEXT.md；如果用户拒绝某个候选方案且理由有价值，主动提议记录为 ADR——架构决策的知识沉淀是内嵌在工作流里的，而不是事后补充。整个技能是一个"探索 → 候选 → 深挖 → 沉淀"的闭环，每一步都有明确的输出物和退出条件。
