## setup-pre-commit：把代码质量门禁装进提交流程

这个技能把三件事打包成一个安装动作：Husky pre-commit hook、lint-staged + Prettier 格式化、typecheck 和测试。它的目标是让"提交前自动检查"从一个需要手动配置的繁琐任务，变成一个可以一键完成的标准化流程。

### 技能定位

触发场景是用户想给项目加 pre-commit 检查，或者想配置 Husky、lint-staged、Prettier 中的任意一个。技能把这三者作为一个整体来安装，因为它们在实践中几乎总是一起出现——单独配置其中一个而不配置其他两个，往往意味着配置不完整。

### 核心设计哲学：自适应而非强制统一

这个技能最显著的设计特征是**在多个维度上做自适应**，而不是假设所有项目都一样：

**包管理器自适应**：通过检测锁文件（`package-lock.json`/`pnpm-lock.yaml`/`yarn.lock`/`bun.lockb`）判断项目使用哪个包管理器，然后在 pre-commit hook 里使用对应的命令。这个细节很重要——在 pnpm 项目里写 `npm run typecheck` 会让人困惑，也可能在某些环境下出错。

**脚本存在性检查**：安装前先检查 `package.json` 里是否有 `typecheck` 和 `test` 脚本，没有的话直接省略对应的 hook 行，并告知用户。这避免了"pre-commit hook 因为找不到脚本而报错"的尴尬——一个失败的 pre-commit hook 比没有 hook 更糟糕，因为它会阻断所有提交。

**Prettier 配置幂等性**：只在项目没有 Prettier 配置时才创建 `.prettierrc`，不覆盖已有配置。这是对"技能应该是幂等的"原则的实践——在已有 Prettier 配置的项目上运行这个技能，不会破坏现有的格式化规则。

### 关键机制细节

**lint-staged 的作用范围**是这个技能设计中最值得关注的细节：它只对**暂存区的文件**运行 Prettier，而不是整个项目。这意味着第一次提交不会因为格式化整个仓库而变得极慢，也不会在 diff 里混入大量无关的格式化变更。`prettier --ignore-unknown` 则确保图片、二进制文件等 Prettier 无法处理的文件被静默跳过，而不是报错。

**三层检查的顺序**（lint-staged → typecheck → test）是有意设计的：lint-staged 最快（只处理暂存文件），typecheck 次之，测试最慢。快速失败原则——如果格式化就失败了，不需要等待完整的类型检查和测试。

**Husky v9+ 的无 shebang 设计**是一个版本特定的知识点：旧版 Husky 的 hook 文件需要 `#!/bin/sh` 开头，v9 不需要。技能明确标注这一点，防止用户从旧教程复制 shebang 导致兼容性问题。

**最后一步用提交本身做冒烟测试**是这个技能最聪明的收尾设计：把安装产生的文件变更提交到 git，这次提交会触发刚刚安装的 pre-commit hook，相当于用真实场景验证整个配置是否正常工作。如果 hook 有问题，这次提交会失败，用户立刻就能发现，而不是等到下次真正提交时才踩坑。

### 亮点总结

`setup-pre-commit` 的价值不在于它做了什么（Husky + lint-staged + Prettier 的组合早已是社区标准），而在于它**把所有容易出错的细节都处理掉了**：包管理器检测、脚本存在性检查、配置幂等性、hook 文件的版本差异、用提交做冒烟测试。这些细节单独看都不难，但手动配置时很容易漏掉其中一两个，导致 hook 在某些环境下静默失效。技能把这些细节固化成了一个可靠的安装流程。
