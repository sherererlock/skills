## migrate-to-shoehorn：测试代码里的 `as` 清除行动

这个技能专门处理 TypeScript 测试代码中的一个顽固问题：为了给函数传入"足够用"的假数据，不得不写一堆 `as Type` 或 `as unknown as Type` 来哄骗编译器。`migrate-to-shoehorn` 提供了一条系统化的迁移路径，把这些类型断言替换成语义更清晰的 `@total-typescript/shoehorn` 工具函数。

### 技能定位

它是一个**迁移型**技能，目标明确：找到测试文件里的 `as` 断言，按模式分类，替换成对应的 shoehorn 函数，验证类型检查通过。触发场景是用户提到 shoehorn、想清理测试里的 `as`、或者在写测试时被大型对象的完整构造拖累。

**测试代码专属**是这个技能的硬性边界——shoehorn 绝不能用于生产代码，技能在开头就明确声明这一点，防止用户把"测试里的便利工具"误用到生产路径上。

### 核心设计哲学：语义化替代类型欺骗

`as` 断言在测试中的问题不只是"不优雅"，而是有三个实质性缺陷：它训练开发者养成绕过类型系统的习惯、需要手动指定目标类型（目标类型变了断言不会报错）、`as unknown as Type` 这种双重断言是在用噪音掩盖意图。

shoehorn 的三个函数各自对应一种明确的测试意图：
- **`fromPartial()`**：我只关心这几个字段，其余的不重要——这是最常见的场景
- **`fromAny()`**：我故意传错误类型来测试错误处理——意图显式，且保留了 IDE 自动补全
- **`fromExact()`**：我要确保传入完整对象——可以在需要时换回 `fromPartial`

这三个函数把原本隐藏在 `as` 背后的测试意图**显式化**了。读代码的人不需要猜"这里为什么要 as"，函数名本身就是注释。

### 关键机制细节

**迁移模式的精确映射**是这个技能的核心价值：

`as Type` → `fromPartial()`：适用于"对象属性太多，测试只用到几个"的场景。原来需要构造一个有 20 个字段的 `Request` 对象，现在只传 `{ body: { id: "123" } }`，shoehorn 负责告诉 TypeScript"相信我，这是合法的"。

`as unknown as Type` → `fromAny()`：这个双重断言通常出现在负面测试中——故意传错误数据来验证错误处理逻辑。`fromAny` 保留了 IDE 自动补全（这是 `as unknown as` 做不到的），同时让意图一目了然。

**工作流的两段式设计**体现了"先理解再动手"的原则：第一步是问卷（哪些文件有问题、是大对象场景还是错误测试场景），第二步才是安装和迁移。这个顺序防止了"盲目替换导致语义错误"——`fromPartial` 和 `fromAny` 的选择取决于测试意图，不能只看语法模式机械替换。

**grep 命令**（`grep -r " as [A-Z]" --include="*.test.ts"`）作为发现工具内嵌在工作流中，让"找到所有需要迁移的位置"这一步有了可执行的具体操作，而不是模糊的"检查你的测试文件"。

### 亮点总结

`migrate-to-shoehorn` 最值得借鉴的是它对**迁移模式的精确分类**。它没有说"用 shoehorn 替换 as"，而是把 `as` 的不同使用场景拆解成三种语义，每种语义对应一个专用函数。这种"先分类再替换"的思路，把一个看似简单的查找替换任务，变成了一次对测试意图的系统性梳理——迁移完成后，测试代码不只是"没有 as 了"，而是"每一处假数据的意图都清晰可读了"。
