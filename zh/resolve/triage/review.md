## triage：Issue 状态机的 AI 操作员

这个技能把 Issue 管理抽象成一个**状态机**，让 AI 扮演 Triage 操作员的角色——读取 Issue、判断状态、推进流转、生成规格。它是整个工程技能套件中最复杂的一个，因为它需要同时处理信息收集、代码库探索、状态判断和文档生成四条并行的工作线。

### 技能定位

触发场景覆盖了 Issue 管理的全生命周期：创建 Issue、分类评估、准备 Agent 规格、管理工作流。它的核心用户是项目维护者，通过自然语言指令（"给我看需要关注的"、"把 #42 移到 ready-for-agent"）驱动 AI 完成原本需要手动操作的 Triage 工作。

### 核心设计哲学：状态机 + 角色分离

技能定义了两套正交的标签维度：
- **类别**（bug / enhancement）：描述 Issue 的性质
- **状态**（needs-triage / needs-info / ready-for-agent / ready-for-human / wontfix）：描述 Issue 在流程中的位置

每个 Issue 必须恰好持有一个类别标签和一个状态标签。这个约束看似简单，却是整个 Triage 流程能够自动化的基础——状态是机器可读的，AI 可以根据状态决定下一步动作，而不需要理解 Issue 的全部语义。

**AI 生成内容必须带免责声明**（`> *This was generated by AI during triage.*`）是一个值得注意的设计决策。它不是形式主义，而是在 AI 和人类协作的工作流中保持透明度的必要机制——维护者需要知道哪些内容是 AI 生成的，以便决定是否需要人工复核。

### 关键机制细节

**三桶展示**是 Triage 入口的核心设计：未标记（从未 Triage）、needs-triage（评估中）、needs-info 且报告者有新活动（需要重新评估）。按最旧优先排序，防止老 Issue 被新 Issue 淹没。这三个桶精确对应了"需要维护者注意"的三种情形，没有冗余。

**Bug 的复现步骤**是 Triage 流程中的独特环节：在"审问"（grill）用户之前，先尝试自己复现——读取报告者的步骤、追踪相关代码、运行测试。这个顺序很重要：成功复现后写出的 Agent Brief 质量远高于仅凭描述写出的，而且能避免向报告者问一些自己本可以验证的问题。

**Agent Brief 的耐久性设计**（AGENT-BRIEF.md）是这个技能最精细的部分。核心原则是"耐久性优于精确性"：Issue 可能在 `ready-for-agent` 状态停留数周，期间代码库会变化。因此 Brief 必须：描述接口和行为契约而非文件路径、描述"做什么"而非"怎么做"、每条验收标准都必须独立可验证。文件路径和行号是明确禁止的——它们是最容易过时的信息。

**`.out-of-scope/` 知识库**是防止重复讨论的机制：当一个 enhancement 被标记为 wontfix 时，不只是关闭 Issue，而是把拒绝决策和理由写入 `.out-of-scope/<concept>.md`。下次 Triage 类似请求时，先检查这个目录，直接引用先例，避免维护者重复解释同一个决策。

**快速状态覆盖**体现了对维护者权威的尊重：如果维护者直接说"把 #42 移到 ready-for-agent"，AI 直接执行，不强制走完整的 Triage 流程。这是"工具服务于人"而非"人适应工具"的设计原则。

### 亮点总结

`triage` 最精妙的地方是它把**知识沉淀**内嵌进了日常工作流：每次 Triage 产生的 Agent Brief、Triage Notes、out-of-scope 文档，都是可被后续 Triage 会话复用的结构化知识。它不只是在处理当前的 Issue，而是在持续构建一个关于"这个项目接受什么、拒绝什么、为什么"的知识库。这种"副产品即资产"的设计，让重复性的 Triage 工作产生了超出预期的长期价值。
