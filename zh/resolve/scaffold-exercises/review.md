## scaffold-exercises：课程脚手架的精确规范

这个技能是为特定项目（ai-hero 课程仓库）量身定制的脚手架工具，负责按照严格的目录约定批量创建练习结构，并确保通过项目自带的 lint 检查。它是少数几个"高度项目特化"的技能之一，但其设计思路对任何需要维护大量结构化内容的项目都有参考价值。

### 技能定位

触发场景是课程作者想从一份计划（章节名、练习名、变体类型）快速生成目录骨架。它的输出不是"大致正确的目录结构"，而是**能通过 `pnpm ai-hero-cli internal lint` 的合法结构**——lint 通过是工作流的强制退出条件，不是可选的验证步骤。

### 核心设计哲学：约定即规范，lint 即合同

这个技能的核心设计决策是：**把所有约定都编码进 lint 规则，而不是依赖文档或口头传递**。目录命名格式（`XX-section-name/`、`XX.YY-exercise-name/`）、必须存在的文件（非空 `readme.md`）、禁止存在的文件（`.gitkeep`、`speaker-notes.md`）——这些约定全部由 lint 强制执行。

技能的工作流因此变得非常清晰：创建结构 → 跑 lint → 修错误 → 循环，直到绿灯。AI 不需要记住所有规则，只需要能读懂 lint 错误并修复。这是"让工具做检查，让 AI 做修复"的分工模式。

### 关键机制细节

**三种变体类型**（`problem/`、`solution/`、`explainer/`）对应课程内容的三种角色，默认只创建 `explainer/`——这个默认值体现了对"先搭骨架再填内容"工作流的理解：大多数情况下先把所有章节的 explainer 占位创建好，再逐步补充 problem 和 solution。

**最小合法 readme** 的设计（一个标题行就够）是对"脚手架阶段不需要真实内容"的务实妥协。lint 只检查非空，不检查内容质量，所以最小 readme 是合法的最小代价。这避免了"为了通过 lint 而写假内容"的尴尬。

**`git mv` 而非 `mv`** 的重命名要求是一个细节但重要的约束：课程练习经常需要重新编号（插入新练习导致后续编号顺移），用 `git mv` 保留历史，用普通 `mv` 会让 git 认为是删除+新建，丢失文件历史。技能把这个 Git 最佳实践内嵌进了重命名工作流。

**数字前缀的双层编号**（章节 `XX`，练习 `XX.YY`）是一个精心设计的命名约定：练习编号包含章节编号，使得练习目录名在整个仓库中全局唯一，不需要知道父目录就能定位一个练习。这对课程内容的引用和链接非常友好。

**lint 规则的完整列表**内嵌在技能文档中，而不是让 AI 去猜。这意味着 AI 在创建结构时就能预判哪些情况会触发 lint 错误，而不是等到 lint 跑完才发现问题。

### 亮点总结

`scaffold-exercises` 最值得借鉴的是它把**项目约定的执行权完全交给了工具**。技能文档描述约定，lint 强制约定，AI 负责在两者之间迭代——这个三角关系让约定的维护成本极低：修改约定只需要改 lint 规则，技能文档和 AI 行为会自动跟进。对于任何需要维护大量结构化内容（课程、文档、模板库）的项目，这种"lint 即合同"的模式都是一个值得采用的架构决策。
