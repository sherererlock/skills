## write-a-skill：技能的自我描述

这是整个套件里最具元层次意味的技能——它教 AI 如何创建新技能。它不只是一个模板，而是一套关于"什么是好技能"的完整设计理论。

### 技能定位

触发场景是用户想创建一个新的 AI 技能。它的工作是把一个模糊的需求（"我想让 AI 能帮我管理 Jira"）转化为一个结构良好、可立即使用的技能包。

### 核心设计哲学：description 是技能的门面

这个技能最重要的洞察集中在一句话上：

> The description is **the only thing your agent sees** when deciding which skill to load.

这个认知重塑了技能设计的优先级。SKILL.md 的正文内容再丰富，如果 description 写得模糊，技能就永远不会被触发——因为 AI 在决定"要不要加载这个技能"时，只看 description，不看正文。

"Helps with documents"和"Extract text and tables from PDF files, fill forms, merge documents. Use when working with PDF files or when user mentions PDFs, forms, or document extraction."——这两个例子的对比，精确地说明了 description 的写作标准：**能力描述 + 触发条件**，缺一不可。

### 关键机制细节

**三步工作流**（收集需求 → 起草技能 → 与用户审查）是一个标准的迭代设计流程，但每一步都有具体的操作内容，不是空洞的流程标签。"收集需求"阶段的四个问题（任务领域、具体用例、是否需要脚本、是否有参考材料）覆盖了技能设计的核心决策点，让 AI 在动手之前就能判断技能的复杂度和结构。

**"何时添加脚本"的判断标准**是这个技能里最有工程价值的部分：操作是确定性的、同样的代码会被反复生成、错误需要显式处理——满足这三个条件之一就应该把逻辑固化成脚本，而不是每次让 AI 重新生成。这个标准背后的逻辑是：**可预测的操作不应该依赖 AI 的即兴发挥**，脚本比生成代码更可靠，也更省 token。

**"何时拆分文件"的判断标准**同样精确：SKILL.md 超过 100 行、内容有明显不同的领域、高级功能很少被用到。100 行是一个具体的阈值，不是"太长了就拆"这种模糊建议。这个阈值背后的考量是上下文窗口效率——每次加载技能都会把 SKILL.md 注入上下文，如果 SKILL.md 过长，大量不相关的内容会占用宝贵的上下文空间。

**Review Checklist** 是这个技能的收尾机制，六个检查项覆盖了技能质量的关键维度：触发词、长度控制、时效性、术语一致性、具体示例、引用深度。其中"No time-sensitive info"（不包含时效性信息）是一个容易被忽视但重要的约束——技能是长期使用的，包含版本号、日期、当前价格等时效性信息的技能会很快过时。

### 亮点总结

`write-a-skill` 最值得借鉴的是它把**技能设计的隐性知识显式化**了。description 的写作标准、脚本的添加时机、文件的拆分阈值、Review Checklist——这些都是有经验的技能作者在实践中积累的判断，但通常以"感觉"的形式存在，难以传递。这个技能把这些判断转化成了可操作的规则，让任何人（包括 AI 本身）都能按图索骥地创建出质量合格的技能。它是整个套件的元技能，也是理解这套技能设计哲学的最佳入口。
