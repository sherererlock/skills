## to-issues：把计划变成可独立执行的 Issue

这个技能做一件事：把一份计划、规格或 PRD 拆解成可以被 AI Agent 或人类独立领取的 Issue，并直接发布到项目的 Issue 追踪器。它是 `setup-matt-pocock-skills` 配置好环境之后，工程技能流水线的第一个实质性动作。

### 技能定位

触发场景是用户想把一个想法或文档转化为可执行的工作单元。它的输出不是任务清单，而是真正发布到 Issue 追踪器（GitHub 或本地 Markdown）的 Issue，带有标准模板、依赖关系和 `needs-triage` 标签，直接进入正常的 Triage 流程。

### 核心设计哲学：垂直切片而非水平切片

这是技能最核心的设计决策，也是它与普通任务拆解工具的本质区别。

**水平切片**是按技术层拆分：先做数据库 Schema，再做 API，再做 UI。每一层单独完成，但在所有层都完成之前，没有任何一个切片是可演示的。

**垂直切片（Tracer Bullet）**是按功能路径拆分：每个 Issue 都是一条穿透所有层的细线——Schema + API + UI + 测试一起完成。切片可以很窄，但必须是端到端完整的，完成后可以独立演示或验证。

这个哲学与 `tdd` 技能中的 Tracer Bullet 概念完全一致，两者共享同一套思维模型。

### 关键机制细节

**HITL vs AFK 分类**是这个技能的独特设计。每个 Issue 被标记为：
- **AFK（Away From Keyboard）**：规格完整，AI Agent 可以无人值守地实现并合并
- **HITL（Human In The Loop）**：需要人类介入，比如架构决策或设计评审

技能明确要求"尽量优先 AFK"——这不只是效率考量，而是一种对 Issue 质量的要求：如果一个 Issue 需要人类介入，说明它的规格还不够清晰，应该先把模糊点解决掉（可能需要一个 HITL Issue 专门处理这个决策），再拆出 AFK 的实现 Issue。

**四步问卷**在发布前强制与用户对齐：粒度是否合适、依赖关系是否正确、是否需要合并或拆分、HITL/AFK 标记是否准确。这个确认环节防止了"AI 自以为是地拆了一堆没人认可的 Issue"的常见问题。

**发布顺序**：按依赖顺序发布（先发阻塞者），这样后续 Issue 的"Blocked by"字段可以引用真实的 Issue 编号而不是占位符。这个细节体现了对 Issue 追踪器实际使用体验的考量。

**Issue 模板**结构清晰：父 Issue 引用、端到端行为描述（不是逐层实现说明）、可勾选的验收标准、阻塞关系。模板刻意要求描述"构建什么行为"而非"怎么实现"，与行为测试的哲学一脉相承。

### 亮点总结

`to-issues` 最精妙的地方是它把"垂直切片"这个工程原则**强制编码进了工作流**：不是建议你这样做，而是通过模板结构和 HITL/AFK 分类，让你不得不思考每个 Issue 是否真的端到端完整、是否真的可以独立执行。它把一个需要经验才能做好的工程判断，转化成了一套可重复执行的操作流程。
